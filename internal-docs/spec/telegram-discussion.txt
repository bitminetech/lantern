> Is devnet 2 blocked by leanSig and leanVM using different XMSS implementations? or is this not a problem?

Yes, they are currently different. For 2 reasons: a) Message Encoding and b) Tweaks. The encoding of the leanSig paper would be too much slow in the current form of leanVM, due to use of Big Integers. But alternative encodings, more zk friendly, exist. The question of which one to use, and how to prove it’s secure, should be answered in the coming months. In the meantime, for devnet 2 I think there are two possibilities: I] validate the encoding outside the snark, as suggested by Benedikt in one the previous PQ interop call. The proof size would be linear in the number of XMSS being aggregated, but still much lower than the full XMSS because size(encoding) << size(xmss). II] use a different encoding (it could be the one currently implemented in leanVM, or another one). b) Currently leanVM does not properly handle hash tweaks. But with the recent updates (in particular one merged 3 days ago) it’s much easier to create custom precompile so adding the tweaks should now be feasible in an efficient way. But I am not fully satisfied with this approach so I kept another branch “simple-lean-vm” where I intend to experiment new designs.
