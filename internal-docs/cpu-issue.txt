# Lantern High CPU Usage Issue

## Summary
Lantern was consuming ~800% CPU (8 cores maxed out) when running in lean-quickstart 
against Zeam, while Zeam only used ~0.02% CPU.

## Root Cause Analysis

### 1. Multiple QUIC Sessions Per Peer (Architectural Issue)
When Lantern starts, it makes multiple concurrent dials to the same peer for different 
protocols (gossipsub, identify, req/resp, etc.). Each dial creates a **new QUIC session** 
with its own network thread because c-libp2p lacks connection pooling.

Observed at startup:
- 8+ concurrent "dialing peer" log messages to the same address
- Each dial calls `libp2p_host_dial_protocol_blocking()` which creates a new QUIC connection
- Each QUIC connection calls `picoquic_start_network_thread()` creating a separate thread

### 2. Busy-Wait in picoquic Socket Loop
Each QUIC network thread runs a socket loop in `sockloop.c` that calls `select()`. 
When `delta_t <= 0`, the loop was setting timeout to 0 (non-blocking poll):

```c
if (delta_t <= 0) {
    tv.tv_sec = 0;
    tv.tv_usec = 0;  // Zero timeout = busy-wait!
}
```

With 8+ threads each busy-polling, CPU usage spiked to ~800%.

## Quick Fix Applied (Partial)

Modified `external/c-libp2p/external/picoquic/picoquic/sockloop.c`:

```c
if (delta_t <= 0) {
    /* Use minimum 10ms delay to prevent CPU-spinning busy-wait */
    tv.tv_sec = 0;
    tv.tv_usec = 10000;  // 10ms minimum
}
```

Also updated Windows path with 1ms minimum.

### Result
- CPU: ~800% → ~200% (4x improvement)
- PIDs: 39-41 → 19 (fewer concurrent connections)

## Why 200% Is Still High

The quick fix reduces spinning but doesn't address the root cause:
1. **No connection pooling**: Multiple QUIC connections to same peer = multiple threads
2. **No dial deduplication**: Concurrent dials create duplicate sessions
3. **Thread-per-session model**: picoquic creates one network thread per QUIC context

## Proper Fix: Connection Pooling in c-libp2p

### Why Connection Pooling is the Right Design

1. **Standard libp2p Architecture**: Both go-libp2p and rust-libp2p use a "Swarm" 
   abstraction that owns connections and reuses them for multiple protocol streams.

2. **QUIC is Built for Multiplexing**: QUIC supports multiplexed streams over a single 
   connection. Each protocol (gossipsub, identify, req/resp) should be a stream on the 
   same QUIC connection, not a separate connection.

3. **Solves Multiple Problems**:
   - CPU: 1 thread instead of 8+
   - Memory: 1 connection state instead of 8+
   - Network: Fewer handshakes, shared congestion control
   - Latency: Streams open instantly on existing connections

### Current Flow (Broken)

```
Protocol A calls dial_protocol_blocking(peer) → new QUIC connection → new thread
Protocol B calls dial_protocol_blocking(peer) → new QUIC connection → new thread
Protocol C calls dial_protocol_blocking(peer) → new QUIC connection → new thread
...
Result: 8+ connections, 8+ threads, ~800% CPU
```

### Target Flow (Fixed)

```
Protocol A calls dial_protocol_blocking(peer) → ConnectionPool.get_or_create(peer)
                                              → new QUIC connection → new thread
                                              → open stream for Protocol A

Protocol B calls dial_protocol_blocking(peer) → ConnectionPool.get(peer) [exists!]
                                              → open stream for Protocol B

Protocol C calls dial_protocol_blocking(peer) → ConnectionPool.get(peer) [exists!]
                                              → open stream for Protocol C
...
Result: 1 connection, 1 thread, ~25% CPU (estimated)
```

### Implementation Plan

#### Step 1: Add Connection Pool Data Structure

Location: `external/c-libp2p/src/host/host.c`

```c
typedef struct libp2p_connection_pool {
    pthread_mutex_t lock;
    // Hash map: peer_id -> connection
    struct {
        libp2p_peer_id peer_id;
        libp2p_conn* conn;
        time_t last_used;
    } *entries;
    size_t count;
    size_t capacity;
} libp2p_connection_pool;

// Add to libp2p_host struct:
struct libp2p_host {
    // ... existing fields ...
    libp2p_connection_pool* conn_pool;
};
```

#### Step 2: Modify Dial to Check Pool First

Location: `external/c-libp2p/src/host/host_dial.c`

```c
libp2p_stream* libp2p_host_dial_protocol_blocking(
    libp2p_host* host,
    const libp2p_peer_id* peer_id,
    const char* protocol_id) 
{
    // 1. Check connection pool for existing connection
    pthread_mutex_lock(&host->conn_pool->lock);
    libp2p_conn* conn = connection_pool_get(host->conn_pool, peer_id);
    
    if (conn && connection_is_alive(conn)) {
        // 2a. Reuse existing connection - just open a new stream
        pthread_mutex_unlock(&host->conn_pool->lock);
        return connection_open_stream(conn, protocol_id);
    }
    
    // 2b. No existing connection - dial new one
    // Use in-flight tracking to prevent concurrent dials to same peer
    if (connection_pool_dial_in_progress(host->conn_pool, peer_id)) {
        // Wait for in-flight dial to complete
        conn = connection_pool_wait_for_dial(host->conn_pool, peer_id);
        pthread_mutex_unlock(&host->conn_pool->lock);
        return connection_open_stream(conn, protocol_id);
    }
    
    // Mark dial in progress
    connection_pool_mark_dialing(host->conn_pool, peer_id);
    pthread_mutex_unlock(&host->conn_pool->lock);
    
    // 3. Create new connection (existing quic_dial logic)
    conn = do_dial_and_select(host, peer_id);
    
    // 4. Add to pool
    pthread_mutex_lock(&host->conn_pool->lock);
    connection_pool_add(host->conn_pool, peer_id, conn);
    connection_pool_clear_dialing(host->conn_pool, peer_id);
    pthread_mutex_unlock(&host->conn_pool->lock);
    
    // 5. Open stream on new connection
    return connection_open_stream(conn, protocol_id);
}
```

#### Step 3: Add Connection Lifecycle Management

```c
// Called when connection closes (remote disconnect, error, etc.)
void connection_pool_on_close(libp2p_connection_pool* pool, libp2p_conn* conn) {
    pthread_mutex_lock(&pool->lock);
    connection_pool_remove(pool, conn->peer_id);
    pthread_mutex_unlock(&pool->lock);
}

// Optional: Periodic cleanup of stale connections
void connection_pool_gc(libp2p_connection_pool* pool, time_t max_idle_secs) {
    pthread_mutex_lock(&pool->lock);
    time_t now = time(NULL);
    for (size_t i = 0; i < pool->count; i++) {
        if (now - pool->entries[i].last_used > max_idle_secs) {
            connection_close(pool->entries[i].conn);
            // ... remove from pool ...
        }
    }
    pthread_mutex_unlock(&pool->lock);
}
```

#### Step 4: Handle Incoming Connections

The listener should also add accepted connections to the pool:

Location: `external/c-libp2p/src/protocol/quic/quic_listener.c`

```c
// When accepting a new inbound connection:
void on_connection_accepted(libp2p_host* host, libp2p_conn* conn) {
    pthread_mutex_lock(&host->conn_pool->lock);
    // Check if we already have a connection to this peer
    libp2p_conn* existing = connection_pool_get(host->conn_pool, conn->peer_id);
    if (existing) {
        // Tie-breaker: keep the one with lower peer ID initiator
        // (standard libp2p connection deduplication)
        if (should_keep_new(existing, conn)) {
            connection_close(existing);
            connection_pool_replace(host->conn_pool, conn->peer_id, conn);
        } else {
            connection_close(conn);
        }
    } else {
        connection_pool_add(host->conn_pool, conn->peer_id, conn);
    }
    pthread_mutex_unlock(&host->conn_pool->lock);
}
```
